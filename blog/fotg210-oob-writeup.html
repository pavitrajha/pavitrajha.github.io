One Cable, Three Bugs: Slab Out-of-Bounds in the Linux FOTG210 USB Device Controller

This write-up documents three independent out-of-bounds memory access vulnerabilities in the Linux kernel FOTG210 USB Device Controller driver. The issues stem from missing bounds validation on attacker-controlled USB control request fields, specifically wIndex, which directly influences endpoint indexing.

Overview

The FOTG210 USB device controller processes USB SETUP packets received from a connected host. These packets are fully attacker-controlled in USB peripheral mode. A missing validation of endpoint indices derived from wIndex leads to out-of-bounds access into the fotg210->ep[] array.

The array is defined as: 

#define FOTG210_MAX_NUM_EP 5
struct fotg210_ep *ep[5];

Valid indices are 0–4, but the driver derives indices using: 

ctrl->wIndex & USB_ENDPOINT_NUMBER_MASK   // yields 0–15

No bounds check is performed before indexing, leading to slab out-of-bounds reads and dereferences.

Root Cause

The vulnerability originates from direct use of attacker-controlled wIndex without validation against FOTG210_MAX_NUM_EP. This allows indices beyond the allocated endpoint array to be used as pointers.

Since fotg210->ep resides near the end of the fotg210_udc structure, out-of-bounds indexing transitions into adjacent kernel heap memory and eventually crosses slab boundaries.

Vulnerability 1: clear_feature OOB Read + Dereference

In fotg210_clear_feature(), the endpoint pointer is resolved directly from unchecked index computation: 

ep = fotg210->ep[ctrl->wIndex & USB_ENDPOINT_NUMBER_MASK];

For wIndex >= 5, this results in an out-of-bounds pointer read. The pointer is immediately dereferenced via: 

ep->wedged

This leads to either:

  • Null pointer dereference if OOB memory contains zeroed slab data
  • Adjacent object memory interpretation as struct fotg210_ep, leading to unsafe field access

Vulnerability 2: set_feature OOB Dereference Chain

In fotg210_set_feature(), the same unchecked index is used: 

epnum = ctrl->wIndex & USB_ENDPOINT_NUMBER_MASK;
fotg210_set_epnstall(fotg210->ep[epnum]);

The OOB pointer is later dereferenced inside: 

ep->fotg210
ep->dir_in
ep->epnum

These values are used to compute MMIO register addresses, which results in hardware register access controlled by corrupted kernel heap data.

Vulnerability 3: get_status OOB Read Leading to Information Leak

In fotg210_get_status(), the OOB pointer is used to compute endpoint stall status: 

fotg210_is_epnstall(fotg210->ep[epnum])

This propagates OOB-derived values into: 

fotg210->ep0_data

The result is returned to the USB host, creating a potential information leak channel derived from kernel heap contents.

Threat Model

The attacker is a malicious USB host connected via physical USB or OTG interface. No authentication or privileges are required.

The vulnerability is reachable at device enumeration or during normal USB control transfer handling.

Impact

All three vulnerabilities allow:

  • Kernel slab out-of-bounds read
  • OOB pointer dereference leading to kernel crash
  • Indirect MMIO register manipulation via corrupted struct fields
  • Potential information disclosure via USB response path

Since USB control requests are unauthenticated and device-facing, exploitation requires only physical or peripheral adjacency.

Fix

The root issue is missing bounds validation on wIndex. 

u8 epnum = ctrl->wIndex & USB_ENDPOINT_NUMBER_MASK;

if (epnum >= FOTG210_MAX_NUM_EP) {
    fotg210_request_error(fotg210);
    return;
}

struct fotg210_ep *ep = fotg210->ep[epnum];

All endpoint access must validate against FOTG210_MAX_NUM_EP before dereferencing.

Affected Versions

All Linux kernel versions containing the FOTG210 UDC driver since initial introduction.

Notes

This issue demonstrates a recurring pattern in USB controller drivers: attacker-controlled control-plane fields being used directly as kernel pointers without validation.

Similar classes of bugs frequently lead to:

  • Kernel memory corruption
  • Heap spraying via USB descriptors
  • Device-side privilege escalation chains
